# 조회 버튼 재클릭 시 API 미호출 — 회사 현장 점검 체크리스트

> **증상 요약**  
> 조회 버튼 1회차 → 10건 조회 성공  
> 같은 조건으로 조회 버튼 2회차 → 화면 변화 없음, **백엔드 컨트롤러 로그도 안 찍힘**  
> 검색 조건 하나만 바꾸고 조회 → 다시 정상 조회됨  

> **참고**  
> 이 문서는 `react_beginner` 학습용 B001 템플릿을 기준으로 작성했습니다.  
> 회사 코드가 완전 동일하지 않아도, **따라 하다 흔히 바뀌는 패턴**을 중심으로 정리했습니다.

## 1. 먼저 할 일 (5분 진단)

현장에서 아래 순서로 확인하면 **원인 범위를 바로 좁힐 수 있습니다.**

### Step A — 브라우저 Network 탭

1. F12 → **Network** 탭 열기
2. 조회 1회 클릭 → `search` (또는 회사 API URL) 요청 확인
3. **같은 조건**으로 조회 2회 클릭

| 2회차 Network 결과 | 의미 | 다음에 볼 곳 |
|---|---|---|
| **요청 자체가 없음** | 프론트에서 API 호출 전에 막힘 | [섹션 2](#2-프론트엔드--api-호출-전에-막히는-경우-가장-흔함) |
| 요청은 있는데 **(disk cache)** / **304** | 브라우저·프록시 GET 캐시 | [섹션 4](#4-http-get-캐시-문제) |
| 요청은 있는데 **서버 로그 없음** | 게이트웨이·캐시 서버·다른 인스턴스 | [섹션 4](#4-http-get-캐시-문제), [섹션 5](#5-백엔드--인프라) |
| 요청 + 서버 로그 둘 다 있음 | API는 탔는데 **화면만 안 바뀜** | [섹션 3](#3-api는-타는데-화면만-안-바뀌는-경우) |

### Step B — console.log 3곳 찍기

회사 코드에서 아래 위치에 로그를 임시로 넣고 2회 클릭해 보세요.

```javascript
// ① SearchPanel 또는 조회 버튼 onClick
onClick={() => {
  console.log("[1] 조회 버튼 클릭");
  onSearch();

// ② Page의 handleSearch
const handleSearch = async () => {
  console.log("[2] handleSearch", getSearchParams());

// ③ useGrid의 search 또는 fetchGridData 직전
const search = async (params) => {
  console.log("[3] search 호출", params);

};
```

|---|---|
| [1]만 안 찍힘 | 버튼 `disabled`, `onClick` 미연결, 다른 요소가 클릭 가로챔 |

## 2. 프론트엔드 — API 호출 전에 막히는 경우 (가장 흔함)

> **핵심 패턴**  
> "검색 조건이 **바뀔 때만** 조회한다"는 로직이 어딘가에 들어가 있으면  
> 지금 증상과 **100% 일치**합니다.

### ✅ 체크 2-1. `useEffect`로 그리드 조회를 걸어두었는가?

**학습 템플릿 (정상 패턴)**  
- `useEffect` → 필터 **옵션**(부서/팀 목록) 로드에만 사용  
- 그리드 **목록 조회** → 조회 버튼 `onClick`에서만 실행  

```javascript
// ✅ 템플릿 — 버튼 클릭 시에만 조회
const handleSearch = async () => {
  await search(getSearchParams());
};
```

```javascript
// ❌ 회사에서 자주 잘못 바꾸는 패턴
useEffect(() => {
  search(getSearchParams());
}, [filters]);  // filters 안 바뀌면 재조회 안 됨!
```

**점검 방법**
- [ ] `useEffect` 안에 `search`, `fetchGridData`, `axios.get('...search')` 가 있는지 검색
- [ ] 의존 배열에 `filters`, `searchParams`, `departmentCode` 등이 있는지 확인
- [ ] 조회 버튼이 `setFilters`만 하고, 실제 API는 effect에만 맡겨져 있는지 확인

**수정 방향**
- 그리드 조회는 **버튼 onClick → handleSearch → search()** 로만 실행
- `useEffect`는 옵션 API(부서/팀/카테고리 등) 로드에만 사용

### ✅ 체크 2-2. React Query / TanStack Query / SWR 사용?

회사 프로젝트에 이미 붙어 있으면, 학습 템플릿의 `useState + axios` 패턴 대신 이걸 썼을 수 있습니다.

```javascript
// ❌ queryKey가 같으면 서버 재요청 안 함
const { data } = useQuery({
  queryKey: ['grid', searchParams],
  queryFn: () => fetchGridData(searchParams),
});
```

**점검 방법**
- [ ] `useQuery`, `useSWR`, `react-query`, `@tanstack/react-query` import 있는지 검색
- [ ] `queryKey` / `key`가 조회 파라미터 기반인지 확인
- [ ] `staleTime`, `cacheTime`, `gcTime`, `refetchOnMount` 설정 확인
- [ ] 조회 버튼이 `refetch()`를 호출하는지 확인

**수정 방향**

```javascript
// 조회 버튼에서 명시적 refetch
const handleSearch = async () => {
  setSearchParams(getSearchParams()); // key 변경이 필요한 경우
  await refetch();
};

// 또는 버튼마다 무조건 서버 호출
const { refetch } = useQuery({
  queryKey: ['grid', searchParams],
  queryFn: () => fetchGridData(searchParams),
  enabled: false,  // 자동 조회 끄고 버튼으로만
});
```

### ✅ 체크 2-3. "같은 조건이면 스킵" 로직이 있는가?

중복 요청 방지를 넣다가 생기는 경우가 많습니다.

**검색 키워드 (회사 코드 전체 검색)**

```
lastParams
prevParams
prevSearch
isSame
JSON.stringify
shallowEqual
deepEqual
skip
duplicate
```

**흔한 패턴**

```javascript
// ❌
const search = async (params) => {
  if (JSON.stringify(params) === JSON.stringify(lastParamsRef.current)) {
    return;
  }
  lastParamsRef.current = params;
  await fetchGridData(params);
};
```

**점검 방법**
- [ ] `useGrid` / `search` 함수 / 공통 API 유틸에 위 키워드 검색
- [ ] `useRef`로 이전 파라미터를 저장하는 코드 있는지 확인

**수정 방향**
- 동일 조건 재조회가 필요하면 **비교 로직 제거**
- 또는 버튼마다 `_t: Date.now()` 같은 값을 파라미터에 추가 (임시) / `refetch` 사용

### ✅ 체크 2-4. 조회 버튼이 실제로 클릭되지 않는 경우

**학습 템플릿**

```jsx
<button
  type="button"           // ← form 안에 있으면 submit 방지
  onClick={onSearch}
  disabled={loading || gridLoading}
>
  조회
</button>
```

**점검 방법**
- [ ] `type="submit"` 인데 `form` 안에 있지 않은지 (또는 반대로 form submit으로 페이지 리로드)
- [ ] `disabled={loading || gridLoading}` 에서 `loading`이 `false`로 안 돌아오는지
- [ ] 첫 조회 후 `gridLoading`이 `true`로 stuck 되지 않는지
- [ ] `finally { setLoading(false) }` 가 있는지 (에러 시에도 loading 해제)
- [ ] 버튼 위에 투명 레이어·다른 div가 덮여 클릭이 안 되는지 (Elements 탭)
- [ ] `onSearch` prop이 Page → Panel까지 제대로 전달되는지

**loading stuck 확인**

```javascript
// useGrid.search — finally 없으면 2번째부터 버튼 disabled
try {
  ...
} finally {
  setLoading(false);  // ← 이게 있어야 함
}
```

### ✅ 체크 2-6. `getSearchParams()`가 매번 같은 객체를 반환?

배열/객체 참조 비교로 스킵하는 코드와 함께 쓰이면 문제가 됩니다.

**학습 템플릿**

```javascript
const getSearchParams = () => ({
  week: filters.week,
  departmentCode: filters.departmentCode,
  sectionCode: filters.sectionCode,
  categoryCodes: filters.categoryCodes.join(","),
  statusCode: filters.statusCode,
  typeCodes: filters.typeCodes.join(","),
});
```

**점검 방법**
- [ ] `categoryCodes`, `typeCodes`를 배열 그대로 넘기다가 직렬화 비교가 꼬이지 않았는지
- [ ] 빈 배열 `[]` → `""` 변환이 1·2회차에 동일한지 (`join(",")` 사용 여부)
- [ ] `undefined` / `null` / `""` 혼용으로 서버·클라이언트 비교가 달라지지 않았는지

### ✅ 체크 2-7. axios 공통 유틸 / 인터셉터에서 요청을 막는가?

회사 `axiosUtil`, `apiClient`, `http` 모듈을 확인하세요.

**점검 방법**
- [ ] request 인터셉터에 동일 URL 스킵 로직 있는지
- [ ] 요청 deduplication / pending map (`Map<url, Promise>`) 있는지
- [ ] debounce / throttle이 조회 버튼에 걸려 있는지
- [ ] `CancelToken` / `AbortController`로 이전 요청만 취소하고 새 요청을 안 보내는지

**학습 템플릿 axiosUtil** — 인터셉터는 response만 처리, 요청 스킵 없음:

```javascript
axiosInstance.interceptors.response.use(
  (response) => response.data,
  (error) => Promise.reject(error)
);
```

## 4. HTTP GET 캐시 문제

학습 템플릿은 조회 API를 **GET**으로 호출합니다.

```javascript
axiosUtil.get("api/grid/search", searchData, {});
```

**같은 URL + 같은 쿼리스트링**이면 브라우저·회사 프록시가 캐시해서 **서버까지 안 갈 수 있습니다.**

### ✅ 체크 4-1. Network 탭 캐시 표시

- [ ] 2회차 Size 컬럼에 `(disk cache)` / `(memory cache)` 표시
- [ ] Status `304 Not Modified`

### ✅ 체크 4-2. 서버 Cache-Control 헤더

- [ ] Spring / nginx에서 GET `/search`에 `Cache-Control: max-age=...` 있는지

### ✅ 수정 방향 (택 1)

```javascript
// 1) 캐시 방지 헤더
axiosUtil.get(url, params, {
  headers: { "Cache-Control": "no-cache", Pragma: "no-cache" },
});

// 2) 쿼리스트링에 타임스탬프 (임시 진단용)
params: { ...searchData, _t: Date.now() }

// 3) 조회만 POST로 변경 (회사 API 규칙에 맞으면)
```

## 6. 학습 템플릿 vs 회사 코드 — 구조 대조표

회사 PC에서 아래 항목을 옆에 두고 비교하세요.

| 항목 | 학습 템플릿 (정상) | 회사 코드에서 확인 |
|---|---|---|
| 조회 트리거 | `onClick` → `handleSearch` | `useEffect`에 있지 않은지 |
| 조회 함수 | `useGrid.search(params)` | 공통 hook / React Query로 대체됐는지 |
| API 호출 | `fetchGridData(params)` 매번 실행 | early return / cache 없는지 |
| HTTP 메서드 | GET `api/grid/search` | GET 캐시 영향 있는지 |
| 버튼 type | `type="button"` | `submit`으로 form 전송하지 않는지 |
| 버튼 disabled | `loading \|\| gridLoading` | 1회 조회 후 stuck 없는지 |
| 파라미터 | `getSearchParams()` 매 클릭 호출 | 배열 join / 빈값 처리 동일한지 |
| 화면 로드 시 | 그리드 **미조회** | 마운트 시 자동 조회 + 버튼 이중 구조 아닌지 |


**이 흐름에서는 조건이 같아도 버튼을 누를 때마다 컨트롤러가 호출됩니다.**  
회사 코드가 이 흐름 중간에 끊기는 지점이 어디인지만 찾으면 됩니다.
